Key Rules (Microsoft Project / VBA)

These rules avoid common runtime issues when working inside an .mpt environment.

Project Application Object

Never use (inside .mpt):

Dim pjApp
Set pjApp = CreateObject("MSProject.Application")  ' ❌ Avoid

Always prefer:

Dim pjApp As Application
Set pjApp = Application  ' ✅ Use host application

General Conventions

Production Macros Scope

  • Production builds only include macros sourced from /macros/import/, /macros/export/, /macros/reports/, and /macros/utils/ in the shipped .mpt.
  • Experimental or test macros must never be placed in these production directories or committed into production builds.

UI Injection Strategy

  • The Plano command is created dynamically via VBA using legacy CommandBars.
  • The command appears under the Add-Ins tab.
  • No Ribbon XML customization is used in the current architecture.

Single Entry Point Rule

  • All runtime behavior must originate from ThisProject.Project_Open().
  • Extension-based detection (.mpt vs .mpp) controls system mode.

Strict VBA Discipline

  • Use Option Explicit in all modules.
  • Declare explicit types for all variables.
  • Avoid Variant unless strictly necessary.

Application Object Usage

  • Always use the host Application object.
  • Never instantiate a new MS Project instance inside template code.

Template → Project Integrity

  • All required modules and forms must exist inside the .mpt so they propagate automatically when saving to .mpp.

Macro Trust Requirements

  • Document required Trust Center settings for macro execution.
  • The template must fail gracefully if macros are disabled.

Commit Standards

  • Follow Conventional Commits:
    • feat: new functionality
    • fix: bug fixes
    • docs: documentation updates
    • chore: refactoring or maintenance
